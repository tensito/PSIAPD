# Отчет по лабораторной работе №3

## Тема

Разработка мультимодального pipeline для распознавания продуктов на изображении и генерации рецептов по найденным ингредиентам.

## Цель работы

Цель лабораторной работы состоит в реализации практической мультимодальной системы, которая получает на вход фотографию холодильника или продуктовой полки, распознает видимые продукты питания, формирует структурированный список ингредиентов и затем генерирует 2-3 рецепта на русском языке.

В работе также проводится оценка качества распознавания продуктов с использованием ручной разметки и метрик Precision, Recall и F1-score. Дополнительно результаты сравниваются с экспериментом, в котором использовалась связка LLaVA-1.5-7B для анализа изображений и Llama 3.2 3B для генерации рецептов.

## Теоретическая база

Мультимодальная модель, или VLM, принимает на вход изображение и текстовую инструкцию. В данной работе VLM используется для извлечения фактов из фотографии: модель должна назвать только те продукты, которые реально видны на изображении.

Языковая модель, или LLM, работает уже не с изображением, а с подготовленным списком ингредиентов. Такой подход разделяет задачу на два этапа:

1. VLM отвечает за зрительное распознавание.
2. LLM отвечает за текстовую генерацию рецептов.

Разделение pipeline делает систему более управляемой: качество распознавания можно оценивать отдельно от качества рецептов, а ошибки можно отнести либо к этапу vision, либо к этапу text generation.

Основная функция распознавания - `detect_products(image_path)`. Она:

1. формирует prompt для модели;
2. передает изображение и инструкцию в processor;
3. запускает генерацию;
4. декодирует ответ;
5. извлекает JSON со списком продуктов;
6. очищает память GPU после обработки изображения.

Prompt явно требует:

- находить только реально видимые продукты;
- не выдумывать продукты;
- возвращать ответ строго в JSON;
- указывать категорию и confidence.

Ожидаемый формат ответа:

```json
[
  {
    "name": "название продукта на русском",
    "category": "овощи / фрукты / молочные / мясо / рыба / напитки / крупы / соусы / другое",
    "confidence": 0.0
  }
]
```

### Постобработка ответа

Так как генеративная модель может вернуть JSON с ошибками форматирования, в коде реализована дополнительная обработка:

| Функция | Назначение |
| --- | --- |
| `extract_json()` | Пытается извлечь JSON-массив из ответа модели |
| `clean_output()` | Удаляет markdown-разметку и лишние символы |
| `extract_complete_json_objects()` | Извлекает завершенные JSON-объекты даже из оборванного ответа |
| `parse_product_object()` | Парсит один объект продукта через `json.loads` или регулярные выражения |
| `extract_products_from_raw_output()` | Формирует итоговый список продуктов и удаляет дубликаты |
| `products_to_eval_text()` | Преобразует список продуктов в строку для таблицы оценки |

Результаты распознавания собираются в таблицу `pred_df`, где хранятся индекс изображения, путь к файлу, предсказанные продукты, категории, количество найденных продуктов и фрагмент сырого ответа модели.

### Оценка качества

Для ручной проверки создается таблица `eval_template.csv`, в которой пользователь заполняет колонку `true_products`. После загрузки заполненного файла рассчитываются:

- `correct` - верно найденные продукты;
- `missed` - продукты, которые были на фото, но отсутствовали в предсказании;
- `hallucinated` - продукты, которые модель назвала, но которых не было на фото;
- `precision`;
- `recall`;
- `f1`;
- количества предсказанных, реальных, верных, пропущенных и лишних продуктов.

Функция `score_row()` считает метрики для одного изображения, а `build_score_df()` применяет расчет ко всей таблице.

## Метрики оценки

Precision показывает долю правильных предсказаний среди всех продуктов, названных моделью:

```text
Precision = Correct / Predicted
```

Recall показывает долю найденных продуктов среди всех реальных продуктов на изображении:

```text
Recall = Correct / True
```

F1-score объединяет Precision и Recall:

```text
F1 = 2 * Precision * Recall / (Precision + Recall)
```

В лабораторной работе используются средние значения по изображениям, то есть macro-average:

```text
mean_precision = среднее Precision по 19 фото
mean_recall    = среднее Recall по 19 фото
mean_f1        = среднее F1 по 19 фото
```

## Сравнение качества рецептов

В основной лабораторной работе рецепты генерирует `Qwen/Qwen2.5-1.5B-Instruct`. В сравнительном эксперименте рецепты генерировала `Llama 3.2 3B`.

Качество рецептов оценивалось качественно по следующим критериям:

| Критерий | Qwen2.5-1.5B-Instruct | Llama 3.2 3B |
| --- | --- | --- |
| Структура ответа | Более стабильный Markdown-формат: название, продукты, добавки, время, шаги | Формат в целом сохраняется, но содержание менее устойчивое |
| Следование списку ингредиентов | Лучше следует переданному списку, но все равно требует контроля | Часто добавляет продукты или сущности, которых нет в списке |
| Логика приготовления | Средняя: рецепты могут быть простыми, но структура понятна | Ниже: встречаются противоречивые шаги и странные блюда |
| Галлюцинации в рецептах | Возможны, особенно из-за противоречивых пунктов prompt | Более выражены: например, появляются продукты и блюда, которых не было во входных данных |
| Практическая пригодность | Требуется ручная редактура | Требуется существенная ручная редактура |

В результатах Llama 3.2 3B были заметны логические проблемы: например, рецепт с названием, подразумевающим мясной продукт, хотя мяса во входном списке не было; смешивание фруктов, лука и специй без понятной кулинарной цели; добавление капусты и грибов при отсутствии этих продуктов среди распознанных ингредиентов.

В основной лабораторной работе prompt для Qwen2.5-1.5B ограничивает список обязательных продуктов и требует помечать дополнительные продукты как опциональные. Это делает рецепты более контролируемыми. Однако в коде есть пункты prompt, которые можно улучшить: одновременно встречаются требования не выдумывать обязательные ингредиенты и разрешение предлагать продукты вне списка. Для повышения качества рецептов эти формулировки лучше сделать однозначными.

## Анализ причин ошибок

Основные причины низких метрик:

1. Сложные сцены: продукты частично закрыты друг другом.
2. Мелкие объекты плохо различимы на фото.
3. Похожие продукты и упаковки приводят к смешению категорий.
4. Модель иногда называет общий класс вместо конкретного продукта.
5. Модель может добавлять вероятные продукты, которых нет на изображении.
6. Для некоторых изображений видимых продуктов много, поэтому вероятность пропуска возрастает.

Отдельно стоит отметить, что метрики vision-этапа напрямую влияют на качество рецептов. Если VLM пропускает важные продукты или добавляет лишние, LLM получает искаженный список ингредиентов и генерирует менее полезные рецепты.

## Рекомендации по улучшению

Для повышения качества системы рекомендуется:

1. Добавить нормализацию названий продуктов: нижний регистр, удаление `ё`, лемматизацию, словарь синонимов.
2. Ввести справочник канонических продуктовых классов.
3. Разделить оценку на строгую и нормализованную.
4. Уточнить prompt для VLM: просить использовать только базовые названия в единственном числе.
5. Уточнить prompt для LLM: убрать противоречивые пункты, разрешить дополнительные продукты только как опциональные.
6. Использовать confidence threshold для снижения количества лишних предсказаний.
7. Добавить ручную проверку продуктов с низкой уверенностью.
8. Увеличить количество тестовых изображений.
9. Проверить качество рецептов отдельной экспертной оценкой по шкале 1-5.
